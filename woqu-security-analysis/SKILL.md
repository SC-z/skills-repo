# 安全加固项目分析

## 资料规则

只使用以下内部资料：

1. `references/安全加固.md` - 第一优先级，默认权威来源。
2. `http://wiznotes.woqutech.com:8080/` - 第二优先级，历史项目笔记。

不要搜索或引用公网资料。如果两个内部来源都没有可用结论，将该项目列为“未查询到”或“需要人工确认”。不要用模型记忆或通用 Linux 加固经验补齐结论。

如果 `references/安全加固.md` 与 WizNote 笔记结论冲突，以 `references/安全加固.md` 为主，并明确说明历史笔记存在不同处理意见。

## 输入抽取

支持以下输入：

- 纯文本：抽取加固项名称、配置路径、参数、命令、检查标准、期望值、客户整改建议和检查结果。
- Markdown：优先解析标题、表格、列表和代码块，再处理普通段落。
- Excel：检查所有非空 sheet，识别检查项、资产、IP、OS、风险等级、合规结果、检查标准、整改方案、命令输出、备注等语义列。使用只读命令或库检查文件；除非用户要求升级 skill，不要创建内置解析脚本。

将每个客户项目规范化为：

- `原始项目`：客户原文或行摘要。
- `关键词`：项目名、配置路径、参数名、命令、产品名和风险短语。
- `上下文`：资产、产品、版本等信息。

只有在多行指向同一个加固控制项时才合并明显重复项。不同资产或不同检查结果会影响建议时，必须保留区分。

## 比对流程

先查询 `references/安全加固.md`：

- 依次尝试完整项目名、配置路径（如 `/etc/ssh/sshd_config`）、参数名（如 `PermitRootLogin`）、命令或模块名（如 `pam_faillock`、`sysctl`）。
- 记录相关标题、建议标签、命令、影响说明和产品特定风险提示。
- 根据资料措辞将本地命中项分类为 `可以修改`、`不建议修改`、`需要条件修改`、`人工判断` 或 `资料不足`。

仅当本地资料未覆盖、结论不明确，或需要历史项目支撑时查询 WizNote：

- 使用 `http://wiznotes.woqutech.com:8080/?q=<urlencoded keywords>` 搜索。
- 优先组合 `安全加固` 与具体控制项、路径、参数、产品名或客户原文。
- 打开相关 `/notes/<guid>` 详情页，只采用直接涉及安全加固、漏洞整改、扫描发现或加固风险分析的笔记。
- 引用笔记标题、URL，以及可见的项目、客户、日期或更新时间信息。

仅在 WizNote 命中的项目应视为历史经验，不强于内置参考资料。如果只是相似案例，将项目放入 `需要人工确认`。

## 风险处理

以下项目需要标记为人工确认：

- 影响或可能影响 QData、QOne、qdatamgr、qlink、qguard、集群通信、vote 心跳、巡检、add_host/add_lun/add_target 或数据库用户的项目。
- 涉及 SSH root 登录、SSH 算法、sudo 权限、wheel/su 策略、PAM 锁定、内核参数、防火墙/iptables、网络路由、存储/网络接口、服务/软件包删除、第三方软件安装的项目。
- 内部资料结论冲突、只找到旧版本案例、或找到的笔记不是同一控制项的项目。

如果资料只说明需要评估、避免操作或找产品/实施确认，不要把可执行整改命令作为确定方案输出。

## 输出格式

报告使用中文，并写入本地 Markdown 文件。

### 本地报告文件

- 如果用户指定输出路径，写入该路径。
- 如果输入是本地文件，默认输出到输入文件同目录，命名为 `<输入文件名去扩展名>-安全加固分析.md`。
- 如果输入是纯文本，默认输出到当前工作目录，命名为 `安全加固分析-YYYYMMDD-HHMMSS.md`。
- 如果目标文件已存在，不要覆盖；在文件名后追加时间戳或序号。
- 最终回复只做简短总结，并给出 Markdown 报告的绝对路径。

报告文件必须包含：

- 标题：`# 安全加固项目分析报告`
- `## 输入信息`：输入文件或文本来源、分析时间、资料范围。
- `## 汇总`：总项目数、已查询到处理方案数量、未查询到数量、需要人工确认数量。
- `## 已查询到处理方案`
- `## 内部资料库未查询到的加固项目`
- `## 需要人工确认`
- `## 查询记录`：列出使用过的本地关键词和 WizNote 搜索 URL，便于复核。

### 已查询到处理方案

使用表格：

| 客户加固项 | 匹配来源 | 建议处理方案 | 影响/注意事项 | 出处 |
|---|---|---|---|---|

`匹配来源` 必须区分 `安全加固.md` 和 `WizNote`。`出处` 必须包含本地参考标题路径；凡是来自内网 WizNote 资料库的结论，必须附带可点击的 `/notes/<guid>` 笔记链接。

### 内部资料库未查询到的加固项目

使用表格：

| 原始项目 | 已尝试关键词 | 未命中原因 |
|---|---|---|

列出所有无法被 `references/安全加固.md` 或 WizNote 支撑的客户项目。若已搜索 WizNote 但没有直接命中，`未命中原因` 中必须附带至少一个 WizNote 搜索 URL，便于复核。

### 需要人工确认

使用表格：

| 原始项目 | 原因 | 建议确认方向 | 已有出处 |
|---|---|---|---|

列出高风险、结论冲突、产品版本敏感或只找到部分匹配的项目。`已有出处` 中凡是涉及内网 WizNote 资料库，必须附带可点击的笔记链接；不能只写“WizNote”或笔记标题。

## 出处标准

- 引用 `references/安全加固.md` 时使用标题路径，例如 `安全加固.md > 1.1.3 用户登录`。
- 引用 WizNote 时使用指向 `http://wiznotes.woqutech.com:8080/notes/<guid>` 的 Markdown 链接；搜索结果只能作为未命中复核记录，不能替代详情页笔记出处。
- 引用原文要短；长笔记只做摘要，不整段复制。
- 不要引用未打开或未检查过的资料。
